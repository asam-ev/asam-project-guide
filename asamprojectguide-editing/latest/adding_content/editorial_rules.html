Editorial rules

In the ASAM Editorial Guide, editorial rules exist. Workgroup members and Technical Writers shall apply the rules when creating documentation for ASAM. This section contains information on how to create and apply editorial rules.

Using editorial rules

  1. Provide the ASAM Editorial Guide to workgroup members and technical writers when a project starts.

  2. workgroup members should try to apply the editorial rules.

  3. Technical writers may educate workgroup members who do not apply the editorial rules.

  4. Technical writers shall review the content committed by workgroup members as soon as possible.

Applying and creating rules

When creating documentation for ASAM, the following approach is applicable:

  1. Check if an appropriate editorial rule exists in the ASAM Editorial Guide when creating documentation.

    1. If an editorial rule exists, apply the rule to the documentation.

    2. If no editorial rule exists, create a new rule.

In the list of editorial rules, each rule can be assigned one of the following three status tags:

Status tag Description

draft

The draft tag is assigned to editorial rules that have been created and are not approved yet.

approved

The approved tag is assigned to editorial rules the OpenX team agreed on and that shall be applied when crating documentation for ASAM.

rejected

The rejected tag is assigned to editorial rules the OpenX team did not agree on and that shall not be applied when crating documentation for ASAM.

Creating new editorial rules

Start the process of creating new editorial rules when an error in the documentation becomes obvious, for example, when sentences in lists sometimes end with a period and sometimes not.

Adding an editorial rule to rules.csv

The rules.csv file contains a list of all editorial rules.

  • Create a rule ID in rules.csv file if no rule exists.

  • Tag the created editorial rule in the rules.csv file as draft.

Example

ID,Short_Description,Link_to_Project_Guide,Link_to_Test,Status
ASM-1,No fixed versioning,-,-,draft
Column Description

ID

Assign a new unique identifier following the convention ASM-[number], for example, ASM-1.

Short_Description

Add a short description that explains the subject of the editorial rule.

Link_to_Project_Guide

Add a link to the section in the ASAM Editorial Guide that contains information on the editorial rule.

Link_to_Test

Add a link to the file that contains the automatic test.

Status

Assign the status for a new editorial rule, for example, draft.

Adding an editorial rule to the ASAM Editorial Guide

This section contains information on how to add an editorial rule

  1. Create a new section in the ASAM Editorial Guide.

  2. Add a tag to the section following the convention [#sec-[ID]], for example, [#sec-ASM-2].

  3. Brainstorm how to document the specific editorial rule.

  4. Have a look at existing resources:

    1. Check if an editorial rule exists in the ASD-STE100 Specification (ASD Simplified Technical English, ASD-STE100, is a Copyright and a Trade Mark of ASD, Brussels, Belgium).

    2. Check if an editorial rule exists in the Google developer documentation style guide and the Microsoft Writing Style Guide.

  5. Extract useful editorial rules from the existing resources.

  6. Adjust the wording in editorial rules when transferring rules to the ASAM Editorial Guide, for example, Microsoft names subunits of a list items, that are named elements at ASAM.

  7. Draft the section for the editorial rule in the ASAM Editorial Guide.

  8. Have a look if an automatic test exists, for example, for Vale.

  9. Add a link to the section of the editorial rule and, if applicable, a link the file for the automatic test to the rules.csv file.
