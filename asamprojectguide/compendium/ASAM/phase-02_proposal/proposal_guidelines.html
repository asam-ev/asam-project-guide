Proposal Contribution Guidelines

Involved roles

  • WG MEMBER

  • PROJECT LEAD

Phase

Step

1. Content Definition

To write a proposal document, its content shall be defined in a way that every participant can contribute and give feedback.
Participants gather content for sections in issues in the proposal repository and discuss them there. The results can then be transferred to the target document in AsciiDoc form.

1.1. Gather Topics In Issues

Use issues to collect topics and tasks for the proposal writing.
If this is an evolution of an existing ASAM standard, consider collecting and using backlog issues from the previous project first.

The name of the issue shall correspond to the respective section.

Example 1. Topic issue for a proposal’s Executive Summary

1.2. Use Issue Templates

When working with issues, use the provided issue templates in the proposal repository. Each issue template contains a description (as comments) on when to use them and what content to add where.

Example 2. Creating an issue from a template

1.3. Label Issues Correctly

When working with issues, make sure that each issues related to the proposal phase contains the label Proposal. This helps with separating issues from the proposal phase later when transferring content to the standardization repository.

If you use the issue templates provided in the proposal repository, the label may be automatically assigned already.
Example 3. Assigning the proposal label

2. How to contribute to a proposal directly in AsciiDoc files

The most direct way of contributing to a project is by creating and editing the AsciiDoc files. However, in order for a consistent workflow and optimal use of the automation steps set up by ASAM, users must first familiarize themselves with the instructions on GitLab and AsciiDoc and follow them. This ensures that all projects work the same way and deliver the same high level of quality that is expected from an ASAM standard.

Each project proposal has the same base structure which is described and explained in the README.adoc file at root level of the repository. Consider the correct document before adding new content to a file.

Do not edit the main.adoc file directly! All content is included through the files in the folder subchapters.

2.1. The GitLab workflow

The repository-based approach of ASAM for working on documents allows members to work in parallel on the same documents without the risk of overwriting each others changes without notice.

Users must follow the Workflow tutorial for the projects using gitlab.

2.2. Writing content in AsciiDoc

All Simulation domain standards at ASAM are written using the AsciiDoc format. AsciiDoc is a mature, plain-text writing format for authoring notes, articles, documentation, books, ebooks, web pages, slide decks, blog posts, man pages and more.

You write an AsciiDoc document the same way you would write a normal text document. There are no markup tags or weird format notations. AsciiDoc files are designed to be viewed, edited and printed directly or translated to other presentation formats.

Users must follow these guidelines:

  • Writing guide

  • Editorial guide
