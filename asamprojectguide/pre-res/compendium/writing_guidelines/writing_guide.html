Writing guide

TECHNICAL WRITER WG MEMBER

This guide is intended to help you writing content for ASAM projects. It provides you a set of hints and rules that make it easier to contribute concise and homogeneous content and documents to the projects.

If you are using VSCode, see the ASAM recommended Visual Studio Code settings.

1. Language

1.1. Use american english (AE)

  • Use "English (US)" (ISO: en-US) as a language setting in your text editor.

  • Mnemonic: -or, -l, -yze, -er

1.2. Verbs with a special meaning (shall, should, may, can)

  • Use these modal verbs to express the level of obligation:

Table 2. Proper modal verbs

1.3. Keep it short

  • Write short sentences.

  • If sentences get too long, consider splitting them up into shorter sentences.

  • Keep headings short (only one line).

Table 3. Shortness is key

1.4. Use present tense

  • Present tense is easy to understand.

  • Present tense is the preferred tense to express technical facts.

Table 4. Present tense
No Yes

Chapter X describes the details.

1.5. Use active voice

  • Active voice is easier to understand than passive voice.

Table 5. Passive voice and active voice
No Yes

The variable is set by the user.

The user sets the variable.

The car is driven to the garage by me.

I drive the car to the garage.

2. Keep it simple

2.1. Use simple constructs

  • Simple constructs require fewer words and are easier to read.

Table 6. Simple constructs
No Yes

2.2. Do not use abbreviations

  • Some less common abbreviations may confuse the reader. Avoid them.

Table 7. No abbreviations
No Yes

3. Be precise

3.1. Do not use synonyms

  • Stick to the same word for an element or concept.

Table 8. Use words consistently
No Yes

Press the key. After hitting the button, the pop-up appears.

Press the key. After pressing the key, a pop-up appears.

3.2. Avoid words that are not precise

  • Do not use words like "basically", "actually", "very", "practically", …​

3.3. Use 'that' and 'which' correctly

  • Use 'that' and 'which' to express the information level:

    • 'that' adds essential information, no commas are used.

    • 'which' adds information that can be left out, commas are used.

Table 9. Precision matters
No Yes

The rule I follow

The rule that I follow

The vehicle that is red drives down the road.

The vehicle, which is red, drives down the road.

4. Formatting

4.1. Consistent formatting

  • Stick to one spelling for a word that allows several spellings (e-mail, email, E-mail).

  • Stick to one style to present information (steps always as numbered lists).

  • Stick to one formatting for a special purpose (first use of technical term: italic)

4.2. Use templates

  • Adoc-templates are available for common types of topics. Use them to create consistent content.

  • Each template contains instructions on how to use the template and sample text that has to be replaced.

  • Currently available templates:

    • Generic

    • Task

    • UML element

    • Use case

4.3. One sentence per line

  • Insert a line break after each sentence.

  • No manual line breaks inside sentences.

⇒ "Only hit 'enter' after a period ('.'), but make sure to hit 'enter' after a period."

1 This is the first sentence.
2 This is the second sentence.
3 This is the third sentence.

4.4. Use paragraphs

  • Use paragraphs to structure your text.

  • A new thought or statement requires a new paragraph.

  • An empty line creates a new paragraph.

1 This is the first sentence of the first paragraph.
2 This is the second sentence of the first paragraph.
3
4 This is the first sentence of the second paragraph.

4.5. Uppercase and lowercase

  • In headings and captions, only the first word begins with an uppercase letter (sentence-style capitalization).

  • Exception: Acronyms and proper names are spelled as is.

  • Use four space characters per indentation level.

  • Do not use tab characters.

  • Use the following text editor settings:

    • Tab size: 4

    • Insert spaces

4.8. Tags

  • All topics, figures, tables and code examples shall have a unique tag.

  • Sections may also have a tag if a link to the section is needed.

  • Note: If you cannot come up with a document-wide unique id, just add [#tag-uid]. This triggers a pipeline warning and the technical writer can help you.

Examples:
[#top-uid]
[#fig-uid]
[#tab-uid]
[#code-uid]
[#sec-uid]
You can generate a UUID using a generator like https://www.uuidgenerator.net/.

4.9. Non-normative sections

Text sections (paragraphs or blocks of text) in the documentation of a standard are considered normative.

If some text sections should be considered as non-normative and only descriptive (for example, background information or examples), mark this sections deliberately as non-normative.

  • Non-normative sections shall be marked with the user-defined asciidoc role non-normative.

    • Put the role attribute [.non-normative] in front of single paragraphs or blocks to mark them as non-normative.

    • Inside a normative section use [.non-normative]#<text># to mark single text passages <text> as non-normative.

Example for normative and non-normative sections
This is a normative paragraph.

[.non-normative]
This is a descriptive, non-normative paragraph.

====
This is a normative block.
====

[.non-normative]
====
This is a descriptive block.
====

This is another normative paragraph.
This sentence is normative.
[.non-normative]#This sentence is non-normative.#
This sentence is normative again.

  • Roles can be combined using the following syntax: [.<first_role>.<second_role>].

Example with multiple roles
[.important.non-normative]
This is a descriptive, non-normative paragraph that is also marked with the important-role.

  • Id and role attributes can be combined.
    Use the long notation for easier reading: [id=<document_id>, role=<document_role>].

Example with role and id combined
[id=top-doc-unique-id, role=non-normative]
= Document title

This is a descriptive, non-normative document.

  • The actual format and style of non-normative sections depends on the entry in the CSS style sheet file that is used for rendering the text and is not coded in the adoc-file itself.

Example of a CSS entry for non-normative text
.non-normative { font-style: italic; color: darkgreen; background: gold }

5. Additional elements

5.1. Lists

  • Lists are a great way to present lots of similar information in a compact form.

  • Use the same word/sentence structure for all list items.

  • Use numbered lists (Asciidoc '.') for steps that require a specific order.

  • Use bullet points (Asciidoc '*') for enumerations and elements that do not require a specific order.

A numbered list example
. First step
. Second step

  3. Third step

5.2. Figures

Figures, like pictures or drawings, may be used to illustrate complex facts.

  • Preferred format: .svg

  • Figures are stored in a subfolder named images located inside the chapter folder.

  • Each figure shall have:

    • a tag

    • a caption

    • the link to the picture

    • an alternative text

[#fig-uuid]

5.4. Special characters

Sometimes you may need to display a character that is reserved for adoc-formatting. Here is how to do it.

  • Underscore (_)
    Use &lowbar; to display an underscore.

  • Pound sign (#)
    Use &#35; to display a pound sign.

  • Protected space ( )
    Use {nbsp} to define a space character is not used for automatic line breaks.
    This is especially useful for value-unit-pairs like 5{nbsp}m/s (5 m/s) to ensure that value and unit are always in the same line of text.

6. Source code

Consider one of the following possibilities if you want to add source code examples to an adoc topic.

  1. You may use small bits of code as an inline literal directly inside your text.

  2. You may add source code as a fragment from a source code file by reference.

  3. You may copy and paste source code directly as a snippet into the adoc topic.

6.1. Inline literals

For very small bits of code using inline literals is the most convenient way to add source code examples. This type of including source code is especially useful if you want to highlight single code elements like methods or class names.

6.1.1. Changes in the adoc file

Enclose the source code in a pair of pluses surrounded by a pair of backticks, like this:

Call the `+source_example_func()+` for the result.

6.1.2. Result

The previous line looks like this as adoc output:

Call the source_example_func() for the result.

6.2. Source code fragments

You can add source code as a fragment from a source code file if you would like to reference to the source code within the source code file.

There are several advantages of using fragments:

  • You do not need to copy the source code into the documentation.

  • You are referencing to the single point of truth, which is the source code file.

  • If you apply changes to the source code file, these changes are automatically applied to your documentation.

Follow these steps to add source code fragments to your documentation:

  1. Add source comment tags at the beginning and the end of the part of code you want to use in your code block.

  2. Use this tag to include the fragment into the adoc documentation.

The example in the following subchapters uses the tag exampleInclude to add a code fragment.

6.2.1. Changes in the source code file

Tags for the inclusion of source code from a source code file
[...]
// tag::exampleInclude[]    (1) (2)
function lorem(ipsum, dolor = 1) {
    const sit = ipsum == null ? 0 : ipsum.sit;
    dolor = sit - amet(dolor);
    return sit ? consectetur(ipsum, 0, dolor < 0 ? 0 : dolor) : [];
}
// end::exampleInclude[]    (3)
[...]
1 To indicate the start of a tagged region in the source code file, insert a comment line in the source code.
2 Assign a name to the tag directive. In this example, the tag is named exampleInclude.
3 Insert another comment line where you want the tagged region to end.

6.2.2. Changes in the adoc file

[#code-example-code-block-tag]    (1)
.ASAM example code block included by tags    (2)
[source, osc2]    (3)
----    (4)
include::<path-to-file>/examplecode.js[tags=exampleInclude]    (5)
----    (6)
1 Insert an anchor for the code block You can reference this anchor using <<code-example-code-block-tag>>
2 Insert a caption for the code block.
3 Define the source code language for code highlighting. For example, osc2, xml, or java.
4 Insert a code block delimiter.
5 Directly include the source code within the source code file with a tag marking fragment.
6 Insert a code block delimiter.
For more information see the Asciidoctor reference.

6.2.3. Result

This is how the reference looks like inside the adoc file:

ASAM example code block included by tags
function lorem(ipsum, dolor = 1) {
    const sit = ipsum == null ? 0 : ipsum.sit;
    dolor = sit - amet(dolor);

6.3. Source code snippets

You can add source code as a snippet into your adoc documentation if you cannot or if you do not want to reference to the source code file. Be aware that you have to maintain two places (your original code and the snippet in the adoc) in case of future changes.

Follow these steps to use source code as a snippet:

  1. Copy the part of source code you want to use as a snippet.

  2. Paste the snippet into your adoc file.

  3. Insert an anchor, a caption, the highlighting info and delimiters as shown in the following subchapter.

6.3.1. Changes in the adoc file

[#code-example-code-block]    (1)

2 Insert a caption for the code block.

4 Insert a code block delimiter.
5 Insert the code block.
6 Insert a code block delimiter.

6.4. Source code type 'osc2'

Use [source, osc2] as a source block identifier for fragments and snippets (see example) to define the language type as 'osc2'. This enables future automatic syntax highlighting.

7. Review process and draft-attribute

The project uses the adoc attribute draft to steer the review process. This FAQ helps you to manage the review and attribute handling.

7.1. Purpose of the draft-attribute

Q1 What is this draft-attribute for?

New content on the master branch has to be marked with the ifdef::draft[] - endif::draft[] 'feature flag' mechanism. New content in this case means content in adoc files that is not already reviewed and agreed on, like newly merged text from a development branch.

A typical adoc topic looks like this:

  [#top-an-example-topic]
  = Topic heading

  ifdef::draft[]

  Some content that is new or still needs
  to be reviewed and agreed on.

  endif::draft[]

  Some other content that is already reviewed and
  ready for being part of the standard documentation.

When a review is started, a WG-Lead or a member of the CCB removes the draft-mechanism for the part that has to be reviewed in the review merge request.

7.2. Preview configuration

Q2 I cannot see my content in the preview. What is wrong?:: You may experience an empty preview in your editor when editing files that contain the mechanism.

+ This is actually a good sign. The preview is working correctly and hides all parts marked as draft.

+ To make the parts visible again, you have to set the attribute draft for your preview window.

7.2.1. VisualStudioCode with asciidoc extension

  1. In the Asciidoc extension settings go to Preview:Attributes - Edit in settings.json.

  2. In the settings.json add the draft attribute:

    "asciidoc.preview.attributes": {
  "draft": ""
}

  3. Save settings.json

7.2.2. AsciidocFX preview

  1. Go to Settings on the right hand side of the preview

  2. Go to the Preview settings tab

  3. Add an Asciidoctor Attribute by double-clicking an empty attribute line

  4. Add attribute draft with value true

  5. Save the settings (small button on the lower left of the tab!)

  6. Reload the preview

Other tools should have similar settings.

7.2.3. Gitlab preview

The preview in Gitlab intentionally shows the release version of the files, meaning that the parts marked as draft are hidden. You can easily switch off the preview and view the source content by clicking on the display source-icon on top of the file window:

Figure 2. How to switch to source-view in Gitlab

The icon next to it switches back to the preview.

7.3. Pipeline artifacts as reference

Q3 My preview looks fine. Why do I get feedback that something is not formatted correctly?:: Please note that all the previews described in this chapter are just for your convenience.
The only reliable documents for reviews are the html-artifacts generated by the CI-pipeline!
These artifacts are linked at the beginning of review merge requests. You can also download the artifacts directly from the pipeline result page.
